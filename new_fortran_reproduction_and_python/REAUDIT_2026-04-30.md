## 4. Issues that remain

### 4.1 L off-by-one in Fig 2 drivers (material)

`tcrw_fig2_clean.f90` line 102: `integer, parameter :: L = 10`.  Sites
are 0..L-1, so the playground is **10 × 10**.  Authors' $L_A$ for the
same 10 × 10 grid is $L_A = 9$.

The paper's Fig 2 caption says "$L = 10$" in the **authors'**
convention, which means an **11 × 11** grid.  So with the current
$L_F = 10$ the new Fortran reproduces the paper's $L = 9$ panel, not
the $L = 10$ panel.  Cross-check against authors at $L_A = 9$ (above)
will agree to MC noise; cross-check at $L_A = 10$ will disagree
because the box is the wrong size.

**Fix:** one-line change.  Either

```fortran
integer, parameter :: L = 11        ! = authors' L = 10
```

or, if you keep $L=10$, document at the top of `fig2_clean.f90` that
this is paper $L=9$.  Same change in `tcrw_fig2_defects.f90`.

### 4.2 Stale comment in `tcrw_fig3a.f90` header

Lines 18–28 say:

```
!   - Lattice IS the L × L playground.  Walker occupies any site in
!     [0, L-1] × [0, L-1].  No surrounding wall ring.
!   - Edge sites: x == 0 .or. x == L-1 .or. y == 0 .or. y == L-1
!     → n_edge = 4L − 4
!   - Bulk sites: 1 ≤ x ≤ L-2 .and. 1 ≤ x ≤ L-2
!     → n_bulk = (L − 2)²
```

But the actual code (lines 178–181) does

```fortran
L_paper = L_list(iL)             ! 4, 9, 19, 49 — paper labels
L_cur   = L_paper + 1            ! authors' convention: L=N ⇒ (N+1) sites
n_edge  = 4 * L_cur - 4
n_bulk  = (L_cur - 2) ** 2
```

So the driver runs the simulation with **authors' convention** (L=4 in
the paper means 5×5 sites), while the header describes the older
"Fortran-L is site count" convention.  No physics impact — every
written summary row tags the row with `L_paper` (the paper label) and
the cross-check `tcrw_fig3a_crosscheck.py` builds the OBC matrix with
$L_A = L_\text{paper}$ and $(L_A+1)$ sites, which matches.  But the
header comment is misleading and worth a 5-line fix to match the body.

`tcrw_fig3{b,cde,f,g,hij}.f90` should be checked for the same drift.

### 4.3 Cross-check scripts have hardcoded paths from a different sandbox

```python
# tcrw_obc_crosscheck_authors.py:23
for candidate in (
    "/Users/prashantbisht/Documents/Claude/Projects/...",
    "/sessions/elegant-wizardly-einstein/mnt/...",
):
```

Current sandbox is `/sessions/zen-determined-cray/...`.  The script
won't find `ROOT` here, so `ROOT` is silently undefined and the
subsequent `os.path.join(ROOT, "fortran_reproduction", ...)` will raise.

Worse: even on the macOS path it points to `fortran_reproduction/` for
TRW.py, but TRW.py actually lives in `new_fortran_reproduction_and_python/`
now.  So this script is broken on every machine.

Same issue in `tcrw_fortran_vs_exact.py` lines 23–29.

`tcrw_fig3a_crosscheck.py` has the same hardcoded paths but uses a
`HERE` fallback first — it works in any sandbox because
`HERE = os.path.dirname(__file__)` resolves correctly.

**Fix:** in the two broken scripts, replace the hardcoded list with
the same `HERE` fallback:

```python
HERE = os.path.dirname(os.path.abspath(__file__))
ROOT = os.path.dirname(HERE) if os.path.basename(HERE) == \
       "new_fortran_reproduction_and_python" else HERE
NEW = os.path.join(ROOT, "new_fortran_reproduction_and_python")
# and use NEW / "TRW._original_code_by_paperauthors.py"
```
